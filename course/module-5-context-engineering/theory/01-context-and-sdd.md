# Модуль 5: Управление контекстом (Context Engineering)

## 1. Проблема потери контекста

AI-агент не имеет памяти между сессиями. Каждый раз он начинает с нуля. Всё, что не помещается в контекстное окно, для агента не существует.

Без документированных решений каждая новая сессия превращается в археологические раскопки. Агент читает код, пытается угадать архитектуру, делает неверные предположения. Он переизобретает уже принятые решения.

Решение проблемы — **артефакты**: файлы, которые хранят контекст между сессиями. Это не просто документация для людей — это память агента, его справочник, его инструкция.

**Context Engineering** — управление тем, какая информация попадает в контекстное окно агента на каждом шаге работы. Вы определяете:

- **Что загружать в начале** — instructions, memories, конституция проекта
- **Что добавлять по ходу** — результаты поиска, файлы, tool outputs
- **Что сжимать или удалять** — compact / summarization

## 2. Spec-Driven Development

**SDD (Spec-Driven Development)** — методология разработки с AI-агентами, при которой вы вместе с агентом пишете детальную спецификацию задачи, а затем агент реализует её автономно. Подробнее об инструментах SDD (Kiro, spec-kit, Tessl): [SDD Tools (Martin Fowler)](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html).

AI-модели обучены на миллиардах примеров кода. Они отлично переводят спецификацию в код, если она чёткая. Проблемы возникают, когда модель не знает, *какой* код вы хотите. SDD решает это: вы явно говорите, что нужно, а агент разбирается, как это сделать.

> [Диаграмма: SDD-цикл (Spec -> Implement -> Review -> Trace -> Constitution)](../diagrams/sdd-cycle.md)

**Процесс:**

1. Пишете спецификацию
2. Даёте агенту
3. Агент реализует
4. Вы делаете ревью
5. Итерация до готовности
6. Фиксация решений в артефактах

Спецификация может быть короткой (10-15 строк для простого endpoint) или длинной (несколько страниц для сложного модуля).

Пример спецификации для endpoint:

```markdown
## POST /api/auth/login

**Цель:** Аутентификация пользователя по email и паролю.

**Request:**
- Body: `{ "email": string, "password": string }`
- Validation: email валиден, password не пустой

**Response:**
- 200: `{ "token": string, "user": { "id": string, "email": string } }`
- 401: `{ "error": "Invalid credentials" }`
- 400: `{ "error": "Validation failed", "fields": [...] }`

**Паттерны:**
- Использовать bcrypt для проверки пароля
- JWT токен с expiry 24h
- Rate limiting: 5 попыток за 15 минут

**Тесты:**
- Успешный логин с валидными credentials
- Ошибка 401 при неверном пароле
- Ошибка 400 при невалидном email
- Rate limiting после 5 неудачных попыток
```

### 2.1. Автономный промпт как спецификация

Пример из проекта **Enji Fleet** — автономный промпт для агента-разработчика. Полный pipeline из 10 шагов с блокирующими правилами и Progress Table.

**Pipeline:**
1. Настройка окружения (установка инструментов без вопросов)
2. Чтение constitution, decision log, entry.md
3. Создание изолированного git worktree
4. Поиск текущей задачи в Progress Table (статусы: 🔄 in progress, ⬜ not started, ✅ done, ❌ failed)
5. Обновление статуса → 🔄
6. Реализация кода + тесты (TDD)
7. Верификация: goimports, go vet, go test, go build
8. Обновление статуса → ✅ или ❌
9. Заполнение trace файла (инкрементально)
10. Создание PR

**Блокирующие правила:**
> **СТОП и исправь:** тест падает, go vet ошибки, coverage < минимума, build падает
> **ЗАПРЕЩЕНО:** переходить к следующей задаче при ошибках, код без тестов

**Режим работы:**
> **Действуй автономно.** Не спрашивай разрешения на создание файлов, установку инструментов, запуск тестов, исправление ошибок.

### 2.2. Minimum Viable SDD: быстрый старт

Не хотите сразу строить полную иерархию артефактов? Начните с минимума:

1. **Напишите AGENTS.md** — конституция проекта с tech stack, conventions, forbidden actions
2. **После каждой задачи просите агента создать trace** в `docs/traces/` — что сделано, какие решения, что пошло не так
3. **Раз в месяц просматривайте traces** и обновляйте AGENTS.md ценными находками

Это даст ~80% пользы от полного SDD. Спецификации, шаблоны и автоматический reflect-mode добавите, когда почувствуете потребность.
