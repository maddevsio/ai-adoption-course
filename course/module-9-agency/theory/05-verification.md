# Верификация

Когда код пишет агент, ценность разработчика — в верификации. Понимать, правильно ли решена проблема, важнее, чем уметь написать решение самому.

### Три уровня верификации

**Слабый:** "Запустил — работает." Код прошёл `npm start`, страница открылась. Этого недостаточно. Баги в edge cases, утечки памяти, race conditions — всё это не видно на уровне "работает".

**Базовый:** "Прочитал, понял, проверил edge cases." Ты читаешь код агента, понимаешь логику, проверяешь граничные случаи вручную. Что будет с пустым массивом? С null? С миллионом записей?

**Сильный:** "Написал тесты до кода, агент реализует под тесты." Ты формулируешь ожидания в виде тестов. Агент пишет реализацию. Тесты проходят — значит контракт выполнен. Это TDD в связке с агентом.

### Декомпозиция — ключевой навык

Разбей проблему на части, каждую из которых можно проверить изолированно. Модуль 5 учит писать спецификации для этого. Здесь про mindset: если ты не можешь проверить результат — задача слишком большая. Разбей на мелкие.

### Правильные вопросы

Верификация начинается с вопросов:

- "А что будет, если входные данные пустые?"
- "Какие пограничные случаи?"
- "Что может пойти не так?"
- "Как это поведёт себя под нагрузкой?"
- "Что произойдёт при сетевой ошибке?"

Это навык, который нужно развивать. Чем больше ты ревьюишь код (свой и чужой), тем острее чутьё на проблемные места.

### Связь с TDD

Тесты — формализованная верификация. Агент пишет код, тесты проверяют. Но верификация шире тестов:

- **Архитектурная корректность** — код в правильном месте, зависимости направлены правильно
- **Бизнес-логика** — решение соответствует требованиям, а не только техническому контракту
- **Безопасность** — нет SQL-инъекций, XSS, утечек данных

### Аналогия

Архитектор не кладёт кирпичи, но проверяет, что здание не рухнет. Senior engineer ревьюит код джуниора — не переписывает, а указывает на проблемы. С агентом — та же модель. Ты не пишешь код, ты гарантируешь его качество.
