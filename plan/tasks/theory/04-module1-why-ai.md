# Задача 04: Модуль 1 — Зачем разработчику ИИ (теория)

## Цель

Написать текст теоретической части Модуля 1. Задача модуля — мотивировать и откалибровать ожидания.

## Что написать

Текст на ~1100 слов, покрывающий:

1. **Эволюция разработки ПО** (2-3 абзаца):
   - Краткая линия: машинный код → ассемблер → компилируемые языки (C, Pascal) → скриптовые/высокоуровневые (Python, JS) → фреймворки и абстракции → ИИ-ассистенты → агенты → оркестры агентов.
   - Паттерн: каждый переход — рост уровня абстракции. Разработчик всё меньше пишет низкоуровневый код, всё больше описывает *что* нужно, а не *как*. Переход от ассемблера к C — то же самое, что сейчас от ручного кода к агентам.
   - Текущий переход: от "пишу код руками с подсказками" к "описываю задачу — агент реализует — я проверяю". Не замена программиста, а следующий уровень абстракции. Как компилятор не заменил программиста, а дал ему работать на уровень выше.

2. **Контекст 2026**: Что конкретно изменилось. Агенты — не эксперимент, а рабочий инструмент. MCP — стандарт. Мульти-модельность — норма. (2-3 абзаца)

3. **3 кейса**: Использовать кейсы из задачи 02. Для каждого — 2-3 предложения с метриками. (3-5 абзацев)

4. **Модель зрелости**: 5 уровней, показать всю карту:
   - Уровень 1-2: Чат / Копилот (спрашиваешь — копируешь, реагируешь на подсказки)
   - Уровень 3: Один агент (ставишь задачу — проверяешь результат)
   - Уровень 4: Агент + MCP + SDD (расширяешь агента инструментами, работаешь по спецификациям)
   - Уровень 5: Оркестрация (координируешь нескольких агентов параллельно)

   Курс ведёт от уровня 1-2 до уровня 5. Показать путь целиком.
   Источник: `sources/Портрет ИИ зрелости.md`, таблица компетенций. (2-3 абзаца)

   **Дополнительная шкала — Steve Yegge "Welcome to Gas Town"** (упомянуть как более гранулярную):
   8 уровней от Zero AI (автокомплит) до "свой оркестратор". Большинство инженеров сейчас на уровнях 2-3 (IDE agent с YOLO mode). Наша 5-уровневая модель — упрощение для курса, Yegge полезен как ориентир "куда расти дальше".
   Источник: `sources/AI-Assisted Development Adoption.pdf`

5. **Где агенты полезны, где осложнены, где нельзя делегировать** (3-4 абзаца):

   **Агенты полезны** — рутинные задачи с чёткими правилами:
   - CRUD, boilerplate, scaffolding (формы, модели, роуты по шаблону)
   - Тесты по существующему коду (unit, integration по паттерну)
   - Рефакторинг по правилам (переименование, выделение метода, миграция API)
   - Документация: README, API-доки, глоссарии, changelog
   - Миграции данных, контент-генерация (лендинги, шаблонные страницы)
   - Линтинг, форматирование, исправление типовых предупреждений
   - Пример из Mad Devs (Enji Fleet): HR-отдел ставит задачу на бота → флит агентов доводит до продакшена без участия разработчика

   **Агенты осложнены** — работают, но требуют плотного контроля:
   - Архитектурные решения: агент предложит варианты, но выбор за человеком
   - Legacy-код без тестов: агент не знает что сломает
   - Интеграция с внешними API без документации: агент будет галлюцинировать endpoints
   - UX/дизайн-решения: агент не понимает пользователя
   - Бизнес-логика со скрытыми правилами ("по пятницам скидка 5%" — нигде не задокументировано)
   - Работа с большими монорепо (>100K строк): контекстное окно не вмещает

   **Нельзя делегировать** — остаётся за человеком:
   - Финальная приёмка кода и code review (агент = джуниор, нужен ревьюер)
   - Деплой в production (человек нажимает кнопку, человек отвечает)
   - Решения по безопасности и доступам (API-ключи, права, firewall)
   - Выбор архитектуры системы (микросервисы vs монолит — бизнес-решение)
   - Работа с персональными данными клиентов (GDPR, compliance)
   - Определение приоритетов бизнеса (что делать первым)

   **Принцип**: агент делает — человек принимает. Автономность агента растёт с качеством спецификации и покрытия тестами. Без спеки и тестов агент опасен. С хорошей спекой и тестами — агент закрывает 80% рутины.

6. **Честные ожидания**: ИИ помогает в ~60% работы, автоматизирует 0-20%. Архитектура, ревью, решения — за человеком. (1 абзац)

7. **Инфографика**: 2 визуализации:
   - "Эволюция разработки": таймлайн от машинного кода до оркестрации агентов — на каждом этапе что абстрагируется, что остаётся за разработчиком
   - "Путь разработчика в курсе": чат → копилот → агент → MCP/SDD → оркестрация — что меняется на каждом уровне, какой модуль за какой уровень отвечает
   (текстовое ТЗ для дизайнера)
