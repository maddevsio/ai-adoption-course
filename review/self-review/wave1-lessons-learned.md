## Ключевые уроки

### 1. Фактические утверждения требуют проверяемых источников

**Что требует источников:**
- ❌ "Нарушение GDPR/HIPAA может стоить компании миллионы"
- ✅ "Нарушение GDPR может стоить компании [до €20 млн или 4% глобального оборота](https://gdpr.eu/fines/)"

**Критерий:** Если читатель может спросить "Откуда эта цифра?", нужен источник.

### 2. Типичные примеры vs конкретные кейсы

**Разница:**
- "Типичные примеры: fast-json-validator" — обобщенный паттерн, не требует источника
- "Например, в проекте X произошло Y" — конкретный кейс, требует источника

**Урок:** Явно маркировать типичные примеры словами "типичные", "часто встречается", "распространенный паттерн"

### 3. Стартапы > Корпорации для кейс-стади

**Почему:**
- Стартапы прозрачнее в метриках (нужно привлекать инвесторов)
- Корпорации могут манипулировать цифрами (маркетинг, NDA)
- Стартапы ближе к аудитории курса (Mad Devs, bootstrapped команды)

**Примеры:**
- ❌ Rakuten: "79% ускорение" — неясно как измерялось, большой бизнес
- ✅ YC W25: "10% WoW growth всего батча" — заявление CEO YC, публичное
- ✅ Lovable: "$20M ARR за 2 месяца" — fastest growth in EU history, измеримо

### 4. Облачные сервисы vs Self-hosting

**Контекст:** Пользователь ожидал упоминание claude.ai/code/, а не Docker/EC2

**Урок:** Для большинства пользователей облачный сервис — основной путь:
- Не нужно управлять инфраструктурой
- Безопасность из коробки
- Shared sessions для командной работы

Self-hosting (Docker) — для enterprise/специфических кейсов, не дефолт.

### 5. Структура документа: раздел "Источники"

**Паттерн из module-5:**
- Inline ссылки в тексте для конкретных утверждений
- Раздел "Источники" в конце для агрегации всех ссылок
- Группировка по темам (GDPR, Sycophancy, Models documentation)

**Преимущества:**
- Легко проверить источники
- Не перегружает текст ссылками
- Удобно для обновления

### 6. Процесс работы с комментариями к PR

**Что сработало:**
- Конкретные комментарии с примерами ("добавь X", "убери Y")
- Пользователь дал прямые ссылки (ralph-orchestrator, claude.ai/code/)
- Четкая обратная связь без неопределенности

**Что можно улучшить:**
- В следующий раз сразу создать draft PR для review
- Использовать GitHub PR review comments (если доступен gh CLI)

## Заключение

Волна 1 показала важность проверяемых источников для образовательного контента. Переход от крупных корпораций к стартапам в кейс-стади сделал курс более релевантным для целевой аудитории (разработчики в стартапах/аутсорсе). Облачные сервисы (claude.ai/code/) более предпочтительны чем self-hosting для большинства пользователей курса.

**Главный урок:** Любое конкретное утверждение о фактах, цифрах, событиях должно иметь проверяемый источник. Типичные примеры должны быть явно помечены как таковые.
